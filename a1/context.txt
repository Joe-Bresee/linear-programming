Project Context: CSC445 LP Report
I am writing a report for a linear programming course (CSC445) investigating the conjecture: "In terms of performance, the Simplex Method is not a competitive option for solving LPs in practice." My goal is to refute this conjecture using a hybrid experimental + theoretical methodology.
Report Structure:

Introduction — state conjecture and methodology
Theoretical context — chronological narrative through key papers
LP model — formal GPU/CPU resource scheduling problem definition
Experimental setup — instance generation, solvers, measurements
Results — timing tables and plots
Discussion — tie results back to theory, real solver implementations
Conclusion — conjecture refuted

Theoretical narrative (chronological):

Dantzig (1951) — Simplex invented, fast in practice from day one
Bland (1977) — cycling solved via pivoting rules, showing active practical development
Klee & Minty (1972) — exponential worst-case proved, source of the conjecture
Khachiyan (1979) — ellipsoid method, first polynomial algorithm but slow in practice
Karmarkar (1984) — practical interior point method, real competitor, conjecture strengthened
Spielman & Teng (2004) — smoothed analysis, theoretical resolution explaining practical efficiency
Bixby (2002) — empirical evidence that industrial Simplex (CPLEX) is extremely competitive

Experiment:

Domain: GPU/CPU resource scheduling LP (MLOps flavour)
LP model: n jobs, m resources, fractional allocation variables, capacity + demand constraints, minimize weighted completion time
Solvers compared via scipy.optimize.linprog:

revised simplex
highs-ipm (interior point)
highs (dual simplex, industry-grade)


Scale: problem sizes n = 10, 50, 100, 500, 1000 jobs
Measure: solve time and iterations at each size
Expected result: Simplex competitive at small-medium n, IPM competitive at large n — either way Simplex is shown competitive, conjecture refuted

Discussion points:

Real solvers (CPLEX, Gurobi) use Revised Simplex with LU factorization — far from textbook Simplex
Degeneracy handled via perturbation, Bland's rule, steepest edge pricing
Presolve techniques reduce problem size before solving
Industrial Simplex is a fundamentally more powerful tool than the worst-case analysis suggests
